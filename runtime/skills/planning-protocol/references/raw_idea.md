# raw idea

下面是完全保留你原意、严格按你原文语序、并融合你定好的工程化中英文术语的结构化版本，只整理结构，不改思想、不改措辞：
 
 
 
需求工程闭环方法论（结构化整理）
 
一、整体理念与基础逻辑
 
1. 对比其他方案：理念一致，但部分细节不同，传统做法过于繁琐。
2. 开发范式：以测试驱动开发为基础，实现端到端闭环，要求测试与功能完备对应。
3. 层级支撑逻辑：- 函数支撑模块
- 模块支撑功能
- 功能支撑需求
核心亮点在于需求到函数的转换过程。
4. 核心思路：先发散、后收敛- 发散：探索可能性，扩大覆盖面，但会产生无效路径与幻觉；
- 收敛：剔除无效内容，形成可落地的需求。
 
 
 
二、需求处理流程（从 Raw Request 到收敛）
 
1. 原始需求（Raw Request）预处理（A 大模型）
 
用户输入场景与需求，即 Raw Request。
基于假设：用户可能说不清、说不全、隐瞒、撒谎。
处理动作：
 
1. 需求即假设：反问模糊、含糊不清的点；
2. 需求非真理：质疑不符合客观事实的内容。
 
2. 需求发散（大模型畅想）（B）
 
大模型进行方案畅想，属于发散阶段，用于补全用户盲区，扩展可能性空间。
 
3. 生成可执行需求项（Actionable Requirement Item）（C）
 
将畅想内容拆解为具体需求点，形成可评估、可落地的 Actionable Requirement Item。
 
4. 正交过滤与冲突判定（D）
 
进入收敛阶段。
引入新 Agent 客观质疑、反对，剔除不合理需求，进行正交过滤，并用方法论判定冲突需求。

5. 端到端逻辑补全（E）

 
收敛后的需求需支撑完整端到端逻辑链条，因此对缺失部分进行补全，确保依赖关系完整。

6. 方案可行性探测（F）

对收敛后的方案进行探测，生成验证代码做调研，丢弃不可行方案，需求再次收敛。

三、后续收敛与约束评估（Constraint Evaluation）

1. 红蓝对抗
红方抛出各类边界问题进行攻击，蓝方判断能否解决或给出缓解方案，需求继续收敛。
2. 需求评审
评审通过后进入编码、测试环节。
3. 迭代闭环：开发中遇到问题，作为新 Raw Request 重新进入流程，形成完整闭环。

四、约束求解与工程落地

1. 需求转代码本质：约束求解过程
把多种可行方案收敛为策略，输出满足需求的解；不满足则重新走流程。
2. 效果：该方法论能让普通模型也变得高度聪明、专注。
3. 实现映射：将原始需求拆解为各类约束，对应到具体函数实现。
4. 测试特性：迭代过程中测试用例明确。

五、过程固化：演进约束语言

将需求、功能、模块、函数及其他信息，统一封装为一种描述性语言，贯穿整个迭代生命周期，命名为演进约束语言。
其他大模型可直接基于该语言按结构编写代码、快速组装。
将整套闭环流程输入大模型，由模型自动生成提示词，可大幅降低开发工作量。
